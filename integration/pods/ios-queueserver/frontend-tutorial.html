IOS queueserver — frontend quick start

Run the real bluesky-queueserver + bluesky-httpserver (IOS profile) locally and talk to it from a React app. ~5 minutes.

1. Prerequisites

  • Docker + Docker Compose
  • git

2. Run it

git clone https://github.com/NSLS2/ophyd-service.git
cd ophyd-service
git checkout demo/ios-nsls2-queueserver
./integration/pods/ios-queueserver/run_demo.sh

First run is slow (builds the profile env, ~several minutes). When it finishes it prints 145 allowed plans / 100 allowed devices and leaves the server up at http://localhost:60610.

Stop it: docker compose -f integration/pods/ios-queueserver/docker-compose.yaml down -v

3. Where the IOS profile lives

You don't need to save it yourself — the pod clones NSLS2/ios-profile-collection (branch main) into the queueserver image automatically. To use a different repo/branch, rebuild with build args:

docker compose -f integration/pods/ios-queueserver/docker-compose.yaml build \
  --build-arg PROFILE_REPO=https://github.com/NSLS2/ios-profile-collection.git \
  --build-arg PROFILE_BRANCH=main

4. Connect from React

Two values are all the UI needs. Put them in .env:

VITE_QSERVER_URL=http://localhost:60610
VITE_QSERVER_API_KEY=iosdemosecretkey0123456789

Send the key on every request as an Authorization header:

const QS = import.meta.env.VITE_QSERVER_URL;
const KEY = import.meta.env.VITE_QSERVER_API_KEY;
const headers = { Authorization: `ApiKey ${KEY}`, "Content-Type": "application/json" };

// read status
await fetch(`${QS}/api/status`, { headers }).then(r => r.json());

// open the environment (loads the IOS profile)
await fetch(`${QS}/api/environment/open`, { method: "POST", headers });

// queue a scan and run it
await fetch(`${QS}/api/queue/item/add`, { method: "POST", headers,
  body: JSON.stringify({ item: { name: "count", args: [["au_mesh"]], kwargs: { num: 3 }, item_type: "plan" } }) });
await fetch(`${QS}/api/queue/start`, { method: "POST", headers });
Auth: /api/ping and /api/status work without a key; everything else (plans, devices, queue, control) needs the key. Simplest: send it on all requests. The default key above is for local dev only — change QSERVER_HTTP_SERVER_SINGLE_USER_API_KEY in docker-compose.yaml for anything real.

5. API reference

Full interactive docs (try requests in the browser): http://localhost:60610/docs  ·  machine spec: /openapi.json. The endpoints a UI usually needs:

MethodPathPurpose
Status & monitoring
GET/api/statusManager + worker + queue state (poll this)
GET/api/pingLiveness check
GET/api/console_outputRecent RE Manager console text
GET/api/stream_console_outputLive console stream (text/event-stream)
Environment (RE worker)
POST/api/environment/openImport the IOS profile (do this first)
POST/api/environment/closeShut the worker down
Plans & devices (build UI menus from these)
GET/api/plans/allowedAvailable plans + parameter signatures
GET/api/devices/allowedAvailable devices
Queue
GET/api/queue/getCurrent queue + running item
POST/api/queue/item/addAdd a plan
POST/api/queue/item/removeRemove an item
POST/api/queue/item/moveReorder
POST/api/queue/clearEmpty the queue
POST/api/queue/startRun the queue
POST/api/queue/stopStop after current item
Run control
POST/api/re/pausePause the running plan
POST/api/re/resumeResume
POST/api/re/stopStop cleanly
POST/api/re/abortAbort
History
GET/api/history/getCompleted items
POST/api/history/clearClear history

Typical UI flow: environment/open → poll status until worker_environment_exists → load plans/allowed + devices/allowed to build forms → queue/item/add → queue/start → poll status / stream console_output.
